Not connected i Why a separate token. The dashboard never holds the Monitor API key. It authenticates to the RGM server with its own bearer token, kept in sessionStorage for this tab only, so the powerful Monitor credential stays server-side and a leaked tab can't drive Monitor directly. Why it matters: it's the cheapest "is the wiring right?" check — every tool below depends on this same server → Monitor connection, so if Status is green the rest can run.

Tagsi The logic. Monitor has no first-class metadata, so the tagger keeps a local YAML overlay (owner, business unit, criticality, cost center) keyed by group. "Sync from Monitor" reconciles groups that were added or removed — dry-run shows the delta, "Sync & write" persists it. Why dry-run first: a write changes source-of-truth config, and these tags become the filter keys the cost, doctor and alert-config tools key off (e.g. criticality=high), so you see the change before committing it.

monitor-tagger — a metadata overlay (owner, business unit, criticality, cost center) on your groups. These tags become filter keys every tool can use. docs ↗

Annotate timelinei What it does. Posts a timestamped note onto Monitor's timeline, optionally pinned to a specific object. Why it's useful: it turns tribal knowledge ("we deployed at 14:03") into data on the same time axis as the metrics, so when you replay an incident the cause sits right next to the effect. These are manual notes; automated deploy/CI notes arrive via the separate webhook receiver.

monitor-annotate — stamps "what changed?" onto the Monitor timeline. Try it: add a note below, then re-generate the post-mortem in the next section — it appears in the timeline. docs ↗

Automated deploy/CI annotations come from the separate monitor-annotate webhook receiver.

Incident replay (post-mortem)i The logic. Given a window (last 2h, or an explicit from/to), it pulls the alerts, slow queries, backups and annotations from that span and assembles a markdown post-mortem — timeline plus an analysis scaffold. Why: the worst time to reconstruct what happened is right after an incident, so replay does the evidence-gathering for you and the write-up starts ~80% done. It pairs with Annotate — notes you added show up in the timeline here.

monitor-replay — turns an incident window into a pre-filled markdown post-mortem: timeline, alerts, slow queries, backups and an analysis scaffold. docs ↗

License & costi What it computes. Compares used vs total license slots, flags servers that are idle or decommissioned yet still holding a slot, and prices the waste; "Project" estimates the cost of onboarding N more servers against your free slots. Why it matters: SQL Monitor licenses by slot, and slots leak silently as servers are retired — this surfaces reclaimable spend and answers "can we fit the next project without buying more?"

monitor-cost — license utilization & spend audit. Flags wasted slots you can reclaim and projects the cost of onboarding more servers. docs ↗

Run to audit license utilization.

Health check (doctor)i What it checks. Runs a battery of lint rules over the install — silent/never-reporting servers, alerts routed to nobody, stale metrics, decommissioned servers still licensed — and grades each finding error / warning / info. Why: monitoring misconfig is invisible until the night it matters (the alert that never fired). Doctor makes those latent gaps visible up front, the way npm audit does for dependencies.

monitor-doctor — "npm audit for Monitor": lints your install for silent servers, alerts that page nobody, stale metrics and decommissioned servers still holding licenses. docs ↗

Run to audit the installation.

Alert config — plan & applyi The model. Alert thresholds and settings live as version-controlled YAML. "Show plan" diffs that desired config against what's live in Monitor; "Apply" only mutates Monitor after you confirm, and tag filters scope the blast radius. Why like Terraform: GUI-clicked alert config drifts and isn't reviewable — treating it as code makes every change diffable, reviewable and repeatable, and dry-run-by-default prevents a fat-fingered mass edit.

monitor-config — "Terraform for Monitor alerts": alert settings as version-controlled YAML. Shows the diff vs live, then applies it (dry-run until confirmed). Apply once and the next plan is clean. docs ↗

Show the plan to see pending changes.

Logi What it is. An append-only, client-side activity log: every tool action and its result or error is timestamped here. Why: these tools make real changes to Monitor, so an at-a-glance audit trail of what ran and what came back keeps you oriented without opening the browser console.
