Tagsi Why it matters: Monitor has no native metadata, but you often want to act on "everything criticality=high" or "owned by team X". Tags add that overlay (owner, business unit, criticality), and they become the filter keys the doctor and alert-config tools rely on. Because writing changes source-of-truth config, you sync as a dry run first to preview the delta, then "Sync & write" to save it.

monitor-tagger A metadata overlay (owner, business unit, criticality) on your groups. These tags become filter keys every tool can use. docs ↗

i Load tags. Reads monitor-tags.yaml and renders it as a matrix: one row per group, one column per tag key. Why: it's the read-only view of the overlay, so you can see which groups carry which metadata before any tool filters on it. i Sync from Monitor (dry run). Compares the live Monitor group list against the overlay and previews the delta: groups that would be added (live but untagged) or flagged missing (tagged but no longer live). Why dry run: nothing is written, so you see the reconciliation before committing it. i Sync & write. The same reconciliation, but it saves new group stubs back to monitor-tags.yaml. Why it's flagged: it edits source-of-truth config on disk, which is destructive enough to confirm. Afterward, Load tags shows the newly added groups, ready to fill in.

Annotate timelinei Why it matters: when you replay an incident later, the hardest question is "what did we change right before it broke?" Annotations put that human context ("deployed web v1.2.3") on the same timeline as the metrics, so cause sits right next to effect instead of living in someone's memory. These are manual notes; automated deploy/CI notes arrive via the separate webhook receiver.

monitor-annotate Stamps "what changed?" onto the Monitor timeline. Try it: add a note below, then regenerate the post-mortem in the next section, and it appears in the timeline. docs ↗

Automated deploy/CI annotations come from the separate monitor-annotate webhook receiver.

Logi Why it matters: these tools make real changes to Monitor, so you want proof of what ran and what came back. The Log is an append-only, timestamped audit trail of every action and its result or error. It's there at a glance, so you're never guessing whether something took effect, with no browser console required.

Health check (doctor)i
          Why it matters: monitoring misconfig is invisible
            until the night it bites: the alert that paged nobody, the server
            that quietly stopped reporting. Doctor surfaces those latent gaps
            before they cost you an outage: silent servers, alerts
            routed nowhere, stale metrics, and decommissioned-but-licensed
            servers. Each is graded error, warning, or info, the way
            npm audit does for dependencies.

        

          monitor-doctor
          "npm audit for Monitor": lints your install for silent servers,
            alerts that page nobody, stale metrics, and decommissioned servers
            still holding licenses.

Alert config: plan & applyi
          Why it matters: alert settings clicked into a GUI
            drift over time and can't be reviewed or rolled back, so nobody knows
            who changed what threshold, or why. Treating them as
            version-controlled YAML fixes that: "Show plan" previews the diff
            against live, "Apply" only changes Monitor on confirm, and tag
            filters cap the blast radius. Dry-run-by-default stops a fat-fingered
            mass edit from going live.

        

          monitor-config
          "Terraform for Monitor alerts": alert settings as
            version-controlled YAML. Shows the diff against live, then applies it
            (dry run until confirmed). Apply once and the next plan is
            clean.
