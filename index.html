Tagsi Why it matters: Monitor has no native metadata, yet you constantly want to act on "everything criticality=high" or "owned by team X". Tags are that overlay (owner, business unit, criticality, cost center) — and they become the filter keys the cost, doctor and alert-config tools rely on. Because a write changes source-of-truth config, you sync dry-run first to preview the delta, then "Sync & write" to persist it.

monitor-tagger — a metadata overlay (owner, business unit, criticality, cost center) on your groups. These tags become filter keys every tool can use. docs ↗

i Load tags. Reads monitor-tags.yaml and renders it as a matrix — one row per group, one column per tag key. Why: it's the read-only view of the overlay, so you can see exactly which groups carry which metadata before any tool filters on it. i Sync from Monitor (dry run). Compares the live Monitor group list against the overlay and previews the delta — groups that would be added (live but untagged) or flagged missing (tagged but no longer live). Why dry run: nothing is written, so you see the reconciliation before committing it. i Sync & write. The same reconciliation, but it persists new group stubs back to monitor-tags.yaml. Why it's flagged: it edits source-of-truth config on disk — destructive enough to confirm — after which Load tags shows the newly added groups ready to fill in.

Annotate timelinei Why it matters: when you replay an incident later, the hardest question is "what did we change right before it broke?" Annotations put that human context ("deployed web v1.2.3") on the same timeline as the metrics, so cause sits right next to effect instead of living in someone's memory. These are manual notes; automated deploy/CI notes arrive via the separate webhook receiver.

monitor-annotate — stamps "what changed?" onto the Monitor timeline. Try it: add a note below, then re-generate the post-mortem in the next section — it appears in the timeline. docs ↗

Automated deploy/CI annotations come from the separate monitor-annotate webhook receiver.

Incident replay (post-mortem)i Why it matters: the worst time to reconstruct an incident is right after one — memories are fuzzy and the clock is ticking. Replay does the evidence-gathering for you: give it a window (last 2h, or an explicit from/to) and it pulls the alerts, slow queries, backups and annotations into a markdown post-mortem that's ~80% written before you type a word. Pairs with Annotate — notes you added show up in the timeline here.

monitor-replay — turns an incident window into a pre-filled markdown post-mortem: timeline, alerts, slow queries, backups and an analysis scaffold. docs ↗

License & costi Why it matters: SQL Monitor licenses by slot, and slots leak silently as servers are retired — so you keep paying for monitoring nobody uses. This audit flags idle or decommissioned servers still holding a slot, prices the waste you can reclaim, and "Project" answers the budgeting question: can the next N servers fit in free slots, or do we need to buy more?

monitor-cost — license utilization & spend audit. Flags wasted slots you can reclaim and projects the cost of onboarding more servers. docs ↗

Run to audit license utilization.

Health check (doctor)i Why it matters: monitoring misconfig is invisible until the night it bites — the alert that paged nobody, the server that quietly stopped reporting. Doctor surfaces those latent gaps before they cost you an outage: silent servers, alerts routed nowhere, stale metrics, decommissioned-but-licensed servers — each graded error / warning / info, the way npm audit does for dependencies.

monitor-doctor — "npm audit for Monitor": lints your install for silent servers, alerts that page nobody, stale metrics and decommissioned servers still holding licenses. docs ↗

Run to audit the installation.

Alert config — plan & applyi Why it matters: alert settings clicked into a GUI drift over time and can't be reviewed or rolled back — nobody knows who changed what threshold, or why. Treating them as version-controlled YAML fixes that: "Show plan" previews the diff vs live, "Apply" only mutates Monitor on confirm, and tag filters cap the blast radius. Dry-run-by-default keeps a fat-fingered mass edit from going live.

monitor-config — "Terraform for Monitor alerts": alert settings as version-controlled YAML. Shows the diff vs live, then applies it (dry-run until confirmed). Apply once and the next plan is clean. docs ↗

Show the plan to see pending changes.

Logi Why it matters: these tools make real changes to Monitor, so you want proof of what ran and what came back. The Log is an append-only, timestamped audit trail of every action and its result or error — there at a glance, so you're never guessing whether something took effect, no browser console required.
